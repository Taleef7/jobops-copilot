# Database Schema

The schema is defined in `db/migrations/001_core_tables.sql`, extended by the numbered migrations through `db/migrations/009_drop_display_name.sql`, and seeded by `db/seed/sample_jobs.sql`. It is Azure-compatible, idempotent, and does not depend on `pgcrypto`. UUID values are generated by the API or the seed scripts rather than by database extensions.

## Schema Goals

- keep jobs as the CRM source of truth
- store the latest structured analysis for each job
- keep outreach drafts auditable and human-reviewed
- leave room for resume tailoring and weekly reporting
- keep the migration repeatable in Azure and local development

## `jobs`

Primary CRM table for each opportunity.

Important fields:

- `id`: UUID primary key
- `job_url`: optional unique URL for dedupe
- `source`: discovery source such as `manual` or `referral`
- `company`: company name
- `title`: role title
- `location`: role location
- `employment_type`: full-time, contract, and similar labels
- `workplace_type`: `remote`, `hybrid`, `onsite`, or `flexible`
- `date_posted`: optional posted date
- `discovered_at`: when the job entered the CRM
- `description_text`: raw job description
- `status`: CRM pipeline state
- `priority`: `high`, `medium`, or `low`
- `fit_score`: optional numeric score between 0 and 100
- `notes`: internal notes
- `next_action`: next manual or automated step
- `next_action_due`: optional follow-up timestamp
- `created_at` and `updated_at`: timestamps

Constraints and indexes:

- status check constraint for the CRM pipeline values
- priority check constraint for `high`, `medium`, and `low`
- fit score check constraint for `0` to `100`
- indexes on `status`, `priority`, and `company`
- trigger to keep `updated_at` fresh on update

## `job_analysis`

Stores the latest structured understanding of a job.

Important fields:

- `id`: UUID primary key
- `job_id`: foreign key to `jobs.id`
- `required_skills`
- `preferred_skills`
- `matched_skills`
- `missing_skills`
- `ats_keywords`
- `fit_summary`
- `recommended_resume_angle`
- `apply_recommendation`
- `confidence_score`
- `model_used`
- `created_at`

Important behavior:

- `job_id` is unique, so each job keeps one current analysis row.
- all skill lists are stored as `jsonb`
- fit score outputs are validated before being saved

## `outreach`

Stores human-reviewed outreach and follow-up drafts.

Important fields:

- `id`: UUID primary key
- `job_id`: foreign key to `jobs.id`
- `contact_name`
- `contact_role`
- `contact_source`
- `linkedin_url`
- `email`
- `message_type`
- `draft_text`
- `status`
- `gmail_draft_id`
- `created_at`
- `sent_at`
- `follow_up_due`

Constraints and indexes:

- status check constraint for `drafted`, `approved`, `sent`, and `skipped`
- message type check constraint for `recruiter_email`, `linkedin_connection`, `referral_request`, `follow_up`, and `thank_you`
- indexes on `job_id` and `status`

## `resume_versions`

Tracks tailored resume versions for future workflow steps.

Important fields:

- `id`: UUID primary key
- `job_id`: foreign key to `jobs.id`
- `base_resume_file_url`
- `tailored_resume_file_url`
- `change_summary`
- `approved`
- `created_at`

## `weekly_reports`

Stores weekly analytics and strategy outputs for the saved report dashboard.

Important fields:

- `id`: UUID primary key
- `week_start`
- `week_end`
- `jobs_discovered`
- `jobs_shortlisted`
- `jobs_applied`
- `outreach_drafted`
- `outreach_sent`
- `responses_received`
- `interviews`
- `common_missing_skills`
- `recommendations`
- `report_markdown`
- `report_url`
- `created_at`

Behavior and indexes:

- `common_missing_skills` and `recommendations` are stored as `jsonb` arrays.
- `report_markdown` stores the human-readable markdown export body.
- `week_start` and `week_end` are kept unique together so a given weekly window upserts cleanly.
- `created_at` is indexed newest-first so the dashboard can load history quickly.
- `report_url` points to a Blob Storage URL when Azure credentials are configured, or to the API export route during development.

## `user_profiles`

Stores per-user resume and grounding context (added in `db/migrations/004_multitenant.sql`).

Important fields:

- `user_id`: text primary key (the Clerk user id)
- `resume_text`
- `resume_file_name`
- `resume_file_url`
- `profile_text`
- `created_at` and `updated_at`: timestamps

Important behavior:

- Identity (name, avatar, email) is owned by Clerk; the app no longer stores its own display name. The `display_name` column was dropped in `db/migrations/009_drop_display_name.sql`.
- `profile_text` (grounding) and the resume columns are kept.
- trigger to keep `updated_at` fresh on update

## `agent_outputs`

Stores persisted AI-agent outputs (interview prep, research, and skill-gap) per job (added in `db/migrations/008_agent_outputs.sql`).

Important fields:

- `id`: UUID primary key
- `job_id`: foreign key to `jobs.id`, `on delete cascade`
- `user_id`: text (the Clerk user id; a denormalized copy of the job owner for cheap filtering)
- `kind`: `interview_prep`, `research`, or `skill_gap`
- `payload`: `jsonb` output body
- `model_used`
- `created_at`

Constraints and indexes:

- kind check constraint for `interview_prep`, `research`, and `skill_gap`
- `unique (job_id, kind)`, so each job keeps one current output per kind and regenerating upserts it
- indexes on `job_id` and `user_id`

## Seed Data

`db/seed/sample_jobs.sql` inserts three representative jobs with fixed UUIDs and upserts them by `id`. That makes the seed file safe to rerun when the database is reset.

The same seed file also inserts one representative weekly report row so the report dashboard has a stable starter record.

The seed set is intentionally varied:

- an AI automation engineering role
- a workflow operations role
- a solutions consulting role

## Bootstrap Flow

Run the bootstrap script from the repo root after setting `apps/api/.env`:

```bash
npm run db:init --workspace @jobops/api
```

That script applies the schema and then seeds the sample jobs. It is safe to rerun after schema resets or local environment refreshes.
